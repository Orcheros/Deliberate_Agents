# Initiative Status — Dashboard & Reporting

## Objective

Scan all initiative directories across every lifecycle stage, read their STATUS.yaml files, cross-reference ROADMAP.md for tier/phase context, and produce:

1. **TRACKER.md** — auto-generated dashboard replacing manual maintenance (matches existing Deuterophos table format)
2. **ROADMAP.md reconciliation** — flag initiatives that exist in STATUS.yaml but are missing from ROADMAP.md (or vice versa)
3. **Activity timeline** — recent stage transitions sorted by date
4. **Summary statistics** — counts per stage, stale initiatives, blocked items

This skill is read-then-write — it does not modify any STATUS.yaml file or ROADMAP.md. It only reads initiative state and generates reports.

## Instructions

### 1. Identify the Target Project

- Read the assignment to determine the target project
- Locate the project's `initiatives_path` (typically `.documentation/initiatives/`)
- Confirm the path exists
- Read `{initiatives_path}/ROADMAP.md` to build an index of initiative IDs, titles, tier/phase assignments, and dependency notes
- Read `{initiatives_path}/TRACKER.md` (if it exists) to understand the current manually-maintained state

### 2. Walk All Stage Directories

Scan these directories in lifecycle order:

```
backlog/
needs-prd/
needs-architecture/
needs-design-study/
needs-design/
needs-stories/
needs-engineering/
needs-qa/
shipped/
retired/
```

For each directory, list all initiative subdirectories.

### 3. Read STATUS.yaml Files

For each initiative found, read its `STATUS.yaml` and extract:

- `id` — initiative identifier
- `title` — initiative name
- `state` — current lifecycle stage
- `updated_at` — last update date
- `updated_by` — agent or human who last updated
- `reason` — why the last transition happened
- `history[]` — array of past transitions (if present)
- Requirement flags: `requires_architecture`, `requires_design_study`, `requires_design`
- Completion flags: `design_complete`, `engineering_complete`, `qa_passed`

If `STATUS.yaml` is missing for an initiative, record it as `status: unknown` with a warning.

### 4. Generate TRACKER.md

Write to `{initiatives_path}/TRACKER.md` using the existing Deuterophos table format:

```markdown
# Initiative Tracker

> Auto-generated by `/initiative-status` on {date}. Do not edit manually.
>
> For strategic context and tier sequencing, see [ROADMAP.md](ROADMAP.md).

## {Stage Name} ({count})

| ID | Title | Updated | Status |
|---|---|---|---|
| {id} | {title} | {updated_at} | {reason or status detail} |

(Repeat table for each stage that has initiatives, in lifecycle order:
 Backlog, Needs PRD, Needs Architecture, Needs Design Study, Needs Design,
 Needs Stories, Needs Engineering, Needs QA, Shipped, Retired)

## Summary

- **Total:** {n} initiatives
- **Active:** {n} (backlog through needs-qa)
- **Shipped:** {n}
- **Retired:** {n}
- **Stale (>14 days):** {n}
```

### 4a. Reconcile with ROADMAP.md

Cross-reference the STATUS.yaml scan against the ROADMAP.md initiative index:

- **In STATUS.yaml but missing from ROADMAP.md** — likely a new intake not yet added to the roadmap. Flag as "not in roadmap."
- **In ROADMAP.md but no STATUS.yaml found** — may have been removed, absorbed, or never scaffolded. Flag as "roadmap-only."
- **Tier mismatch** — if ROADMAP.md places an initiative in a tier that implies a different priority than its current lifecycle stage, note the discrepancy.

Include reconciliation findings in the detailed report (Step 7), not in TRACKER.md itself.

### 5. Generate Activity Timeline

Collect all transitions from `history[]` arrays and `updated_at` fields. Sort by date descending. Include the 20 most recent entries.

```markdown
## Recent Activity

| Date | Initiative | Transition | By | Reason |
|---|---|---|---|---|
| {date} | {title} | {from} → {to} | {by} | {reason} |
```

### 6. Generate Summary Statistics

Calculate:
- Count of initiatives per stage
- Stale initiatives: `updated_at` more than 14 days ago and not in `shipped/` or `retired/`
- Blocked initiatives: in a stage where a prerequisite flag is unset (e.g., in `needs-qa/` but `engineering_complete: false`)
- Skip-eligible: initiatives where conditional stages will be skipped based on requirement flags

### 7. Write Outputs

- **TRACKER.md** → `{initiatives_path}/TRACKER.md` (overwrites existing)
- **Detailed report** → `.deliberate/reports/{project-slug}/initiative-status-report.md`
  - Includes: dashboard tables + activity timeline + summary stats + stale/blocked warnings + ROADMAP.md reconciliation

## Output

- `{initiatives_path}/TRACKER.md` — auto-generated dashboard (replaces manual version, matches existing table format)
- `.deliberate/reports/{project-slug}/initiative-status-report.md` — detailed report with timeline, statistics, and roadmap reconciliation

## Constraints

- **Read-only on STATUS.yaml** — never modify initiative state; only read and report
- **Idempotent** — running twice produces the same output (timestamps aside)
- **Backward compatible** — initiatives without `history[]` are handled gracefully (timeline shows only the current state from `updated_at`)
- **No git operations** — the caller decides whether to commit the generated reports

## Transition

After generating reports, the caller can:
- Review the dashboard for planning decisions
- Use stale/blocked warnings to prioritize unblocking work
- Run `/initiative-pipeline` on specific initiatives identified in the report
